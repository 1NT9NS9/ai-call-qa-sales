# Security Baseline

## Purpose

This document defines the minimum security requirements for the MVP.
It is a baseline only. It does not define a full production security program.

System design is defined in [ARCHITECTURE.md](ARCHITECTURE.md).
Core data contracts are defined in [CONTRACTS.md](CONTRACTS.md).
Integration boundaries are defined in [INTEGRATIONS.md](INTEGRATIONS.md).
Repository layout is defined in [STRUCTURE.md](STRUCTURE.md).

## Scope

The baseline applies to:

- uploaded audio files
- transcripts
- analysis results
- secrets and environment configuration
- outbound webhook delivery
- logs generated by the pipeline

## Secrets And Configuration

- Secrets shall not be committed to the repository.
- Runtime secrets shall be supplied through environment variables or secret injection at deploy time.
- A local root `.env` file may be used for development only and shall remain git-ignored.
- `.env.example` shall contain placeholders only.
- Provider API keys, database credentials, and webhook secrets shall be loaded through configuration code, not hardcoded in services.

## Audio And Transcript Handling

- Audio files shall be stored in the mounted storage location only.
- The database shall store `audio_storage_key` or relative path, not raw audio blobs.
- Transcript data shall be persisted only for MVP pipeline needs.
- Access to raw audio and transcript data shall be limited to the backend service and explicitly approved integrations.

## Logging Rules

- Logs shall include operational metadata such as `call_id`, stage, latency, and status.
- Logs shall not contain secrets.
- Raw audio content shall never be logged.
- Full transcript text shall not be logged by default.
- Analysis payloads shall be logged only in redacted or summary form when logging is required.

## Webhook Security

- Outbound webhook targets shall be explicitly configured.
- Webhook delivery shall use `HTTPS` outside local development.
- The system shall store delivery status, response code, and error summary, but not sensitive response bodies unless explicitly required.
- If request signing is introduced, signing logic shall remain inside the webhook adapter.

## External Provider Access

- STT, embeddings, and LLM providers shall be accessed through adapter interfaces only.
- Provider credentials shall be isolated from domain and application logic.
- Requests sent to external providers shall contain only the minimum data required for the call.

## Data Exposure Constraints

- Call lifecycle state and analysis review state shall remain separate.
- Demo assets in `data/demo/` shall not contain real sensitive customer data.
- Seed knowledge documents in `data/kb_seed/` shall be safe for development and testing use.

## Local Development Requirements

- Local development shall use non-production credentials.
- Local webhook testing shall target controlled endpoints only.
- Production-like secrets shall not be copied into local documentation or example files.

## Out Of Scope For The MVP

The following are not required as part of the MVP baseline:

- multi-tenant isolation
- billing security controls
- full audit logging
- advanced key rotation workflows
- formal data retention automation
- advanced monitoring or SIEM integration
